Consider removing usage of invalidateTransaction in favor of just throwing
exceptions.
  - May be needed in some contexts but probably ought to be avoided when
    possible
  - Makes it difficult to detect problems and recover from them
  - Hides the source of the error
  - Limited to just a single error per tx setup

Need a mechanism to handle "out of space" errors
  - Will present when we attempt to commit an object update to the store
    - Read-Only or out of space underlying media will cause it to fail
  - Will clog up the CRL if left untreated
  - Might start with just relying on TransferStore to a device with space
    - Might be able to automate this eventually
  - Could set up "spillover stores"
    - Dedicated to hosting data for full stores
    - Lots of implications though, will require some thought

Could add concept of EventHandlers
  - Identified by UUID
  - Optionally able to be attached to "things of interest"
    - Such as an AllocationGroup storage changing by 0.5%
  - When an "event" triggers:
    - Use TypeRegistry to get a factory for handler corresponding to UUID
      - Factory type must castable to a base factory type
        - Ensures that handler objects will have the needed API for handling events
          of that type. like AllocGroupUsageChangeEventHandler

For maximum recovery speed, DurableRecoveryManagerTask
  - Delegates restore operations to groups of workers
  - Workers restore objects
     - Store the results locally (not re-erasure coded)
       - Could use a RocksDB-like backend. Key = ObjId + PoolIndex
     - Update dist datastructure noting their location
  - End-location downloads rebulit slices as they're created

  + End-location can be throttled so as not to overload bandwidth
  + Data can be rebuilt at maximum speed (no choke points)
  + Rebuilt data can be read from dist data structure at a speed penalty
